# RDR-030: Schema Navigation & Discovery — Implementation Plan

## Executive Summary

This plan decomposes RDR-030 into 8 task beads across 4 phases. Phases 1--2 are
P1 priority (schema tree enhancement and field inspector). Phases 3--4 are P3
(diagram visualization and GraphQL introspection). All RDR-030 dependencies are
closed (RDR-026, RDR-027, RDR-020, RDR-029).

Phase 1 is a **delta** on the existing `FieldSelectorPanel` built in RDR-029 ---
not greenfield. Phase 4 carries a HIGH risk around AST round-trip correctness and
requires a spike before implementation.

```
Kramer-dsxh (epic, existing) ── RDR-030: Schema Navigation & Discovery
  |
  |-- Phase 1: FieldSelectorPanel Enhancement (P1)
  |     |-- Kramer-ac5w  T1A: State badges + type/cardinality labels

        |-- Kramer-hlxz  T4B: QueryExpander AST field add/remove (spike + impl)
        |-- Kramer-b9s2  T4C: IntrospectionTreePanel + re-execution pipeline
```

## Critical Path

**P1 work**: `{T1A, T1B, T2A}` all start in parallel --> T2B follows T2A.
Longest chain: T2A --> T2B (2 tasks sequential).

**P3 work**: T3 waits on T1B only.

**P4 work**: T4A --> T4B --> T4C (3 tasks sequential, longest chain in the plan).
The spike in T4B is the critical risk gate.

## Parallelization Opportunities

Three tasks can start immediately with no blockers:
1. **Kramer-ac5w** (T1A) — state badges on tree cells
2. **Kramer-b56f** (T1B) — click-to-scroll navigation

Maximum parallelism: 3 agents on {T1A, T1B, T2A}, plus optionally T4A.

## Phase 1: FieldSelectorPanel Enhancement

### Rationale

`FieldSelectorPanel` from RDR-029 provides the TreeView with visibility
checkboxes and hide-if-empty toggles. Three delta features remain from the
RDR-030 spec: state badges, click-to-scroll, and type/cardinality labels.

### T1A: State Badges + Type/Cardinality Labels (Kramer-ac5w)

**What**: Modify `FieldTreeCell.updateItem()` to render sort arrow, filter
indicator, render mode label, and type/cardinality info per tree node.

**Where**: `FieldSelectorPanel` inner class `FieldTreeCell`,
`kramer/src/main/java/com/chiralbehaviors/layout/query/FieldSelectorPanel.java`

**How**:
- Query `queryState.getFieldState(path)` for `sortFields`, `filterExpression`, `renderMode`
- Sort arrow: unicode glyph or CSS-styled Label when `sortFields` non-null
- Filter indicator: small dot/icon when `filterExpression` non-null
- Render mode label: text badge when `renderMode` overridden
- Type label: "scalar" for `Primitive`, child count for `Relation`
- Right-align badges using `HBox` with `Region` spacer
- Register `queryState.addChangeListener()` to trigger tree refresh on state changes
- New CSS classes: `field-sort-badge`, `field-filter-badge`, `field-render-mode-badge`, `field-type-label`

**Tests**: Extend `FieldSelectorPanelTest.java` with 6 new test methods.

### T1B: Click-to-Scroll Navigation (Kramer-b56f)

**What**: Selecting a tree node scrolls the layout to the corresponding field.

**Where**: `FieldSelectorPanel` (callback), `AutoLayout` (new `scrollToPath()`),
`AutoLayoutController` (wiring)

**How**:
- Add `Consumer<SchemaPath> onFieldSelected` to `FieldSelectorPanel`
- TreeView selection listener resolves path via `resolvePath()`, invokes callback
- New `AutoLayout.scrollToPath(SchemaPath)`: traverse scene graph to find
  `VirtualFlow` with matching `schemaPath`, call `showAsFirst(index)`
- Pattern: inverse of existing `hitSchemaPath()` / `hitSchemaPathRecursive()`
- Wire in `AutoLayoutController.initialize()`:
  `fieldSelectorPanel.setOnFieldSelected(path -> layout.scrollToPath(path))`

**Tests**: Extend `FieldSelectorPanelTest.java`; optionally new `AutoLayout` test.

## Phase 2: Field Properties Inspector

### Rationale

Users need to view and edit all 14 `FieldState` properties for any field. The
existing context menus (RDR-029) expose a subset; the inspector provides complete
access with inline editing.

### T2A: FieldInspectorPanel Property Display and Editing (Kramer-tuiw)

**What**: New JavaFX `VBox` control showing all `FieldState` properties with
inline editors that dispatch `LayoutInteraction` events.

**Where**: New file `kramer/src/main/java/com/chiralbehaviors/layout/query/FieldInspectorPanel.java`

**How**:
- `setSelectedField(SchemaPath, SchemaNode)` populates from `queryState.getFieldState(path)`
- Property row for each of 14 `FieldState` fields:
  - Boolean fields: `CheckBox`
  - String fields: `TextField` + clear button
  - Double fields: `Spinner<Double>` or `TextField`
  - Enum-like fields: `ComboBox`
- Edits dispatch `LayoutInteraction` via `InteractionHandler.apply()`
- SchemaPath breadcrumb at top via `Label` with `path.toString()`
- QueryState change listener refreshes display
- Hide Relation-only fields (hideIfEmpty, collapseDuplicates) for Primitive nodes

**Design decision**: 7 of 14 fields lack dedicated `LayoutInteraction` variants.
Options: (a) add new variants for undo support, (b) use direct setters. Decision
deferred to implementation time — evaluate with sequential-thinking.

**Tests**: New `FieldInspectorPanelTest.java` with 7 test methods.

### T2B: Selection Binding + Controller Integration (Kramer-1797)

**What**: Wire tree selection and layout cell clicks to update the inspector.
Integrate into `AutoLayoutController` sidebar.

**Where**: `AutoLayoutController.initialize()`, `FieldSelectorPanel` (callback)

**How**:
- Tree selection: `treeView.getSelectionModel().selectedItemProperty()` listener
- Layout click: extend existing `MOUSE_CLICKED` handler to update inspector
- Sidebar layout: `SplitPane` with tree (top) + inspector (bottom) in left sidebar
- Toggle: `Cmd+Shift+I` keyboard shortcut

**Tests**: 4 new test methods covering both selection sources and integration.

## Phase 3: Schema Diagram (P3, Wave 10)

### T3: SchemaDiagramView (Kramer-u3io)

**What**: Visual tree diagram of the `Relation`/`Primitive` hierarchy with
click-to-navigate wiring.

**Where**: New file `kramer/src/main/java/com/chiralbehaviors/layout/query/SchemaDiagramView.java`

**How**:
- `Pane` with positioned `Label`/`Region` nodes (not Canvas, need click targets)
- Top-down tree layout: compute subtree widths bottom-up, assign x/y by depth
- Relation nodes as rectangles, Primitive nodes as smaller rectangles, Line edges
- Click handler invokes `Consumer<SchemaPath>` navigate callback
- Reuses `scrollToPath()` from T1B for navigation
- Optional zoom/pan via `ScrollPane` wrapper

**Risk**: Low. Kramer schemas are shallow (2--4 levels). For >50 nodes, add lazy
expansion.

**Tests**: New `SchemaDiagramViewTest.java` with 5 test methods.

## Phase 4: GraphQL Schema Introspection (P3, Wave 10)

### Rationale

Progressive query construction --- the ability to explore available fields from
the GraphQL endpoint and add them to the active query --- is a core capability
from the Bakke papers (CHI 2011, CIDR 2011). This phase carries HIGH risk around
AST modification.

### T4A: TypeIntrospector (Kramer-799f)

**What**: Execute `__schema` introspection query against the GraphQL endpoint and
parse the result into a browseable type tree.

**Where**: New file `kramer-ql/src/main/java/com/chiralbehaviors/layout/graphql/TypeIntrospector.java`

**How**:
- Data model: `IntrospectedType`, `IntrospectedField`, `TypeRef` records
- `introspectionQuery()`: returns `__schema` query string
- `parse(JsonNode)`: parses introspection response into type list
- `typeIndex()`: `Map<String, IntrospectedType>` for lookup
- Filter system types (`__Type`, `__Field`, `__Schema`, etc.)
- Note: name is `TypeIntrospector` (not `SchemaIntrospector`, which is taken)

**Tests**: New `TypeIntrospectorTest.java` with 4 test methods.

### T4B: QueryExpander — Spike + Implementation (Kramer-hlxz)

**What**: AST modification for adding/removing Field nodes in SelectionSets.
Structurally different from `QueryRewriter` (which only adds Arguments).

**Where**: New file `kramer-ql/src/main/java/com/chiralbehaviors/layout/graphql/QueryExpander.java`

**SPIKE REQUIRED**: Before implementation, validate:
1. graphql-java immutable AST modification via `Field.transform()` / `SelectionSet.transform()`
2. Round-trip: modify AST --> `AstPrinter.printAst()` --> re-parse --> structure matches
3. Nested field addition (relation fields need sub-SelectionSets)

**How**:
- `addField(Document, SchemaPath parentPath, String fieldName)`: insert new Field node
- `removeField(Document, SchemaPath fieldPath)`: remove Field from SelectionSet
- `serialize(Document)`: `AstPrinter.printAst()`
- For new relations: create SelectionSet with minimal sub-fields (id + name-like per RF-6)
- Use graphql-java `.transform()` for immutable AST modification

**Risk**: HIGH. If round-trip fails, fallback to string-based query construction.

**Tests**: New `QueryExpanderTest.java` with 6 test methods. Round-trip test is
the critical gate.

### T4C: IntrospectionTreePanel + Re-execution Pipeline (Kramer-b9s2)

**What**: UI panel in explorer module for browsing endpoint types and
adding/removing fields from the active query.

**Where**: New file `explorer/src/main/java/com/chiralbehaviors/layout/explorer/IntrospectionTreePanel.java`

**How**:
- `TreeView` of `IntrospectedType`/`IntrospectedField`
- Per-field "Add" or "Remove" button based on whether field is in current query
- Add action: `QueryExpander.addField()` --> serialize --> re-execute via `fetchAsync`
- Graceful degradation: disabled when introspection unavailable
- Wire into `AutoLayoutController` after `setData()` resolves schema context

**Tests**: 5 test methods covering tree display, add/remove actions, re-execution,
and graceful degradation.

## Risk Summary

| Risk | Severity | Task | Mitigation |
|------|----------|------|------------|
| AST round-trip for field addition | HIGH | T4B | Spike validates approach first; string fallback |
| Large schema trees unwieldy | Medium | T1A, T3 | Lazy loading, zoom/pan |
| FieldState fields without LayoutInteraction variants | Medium | T2A | Decision at impl time: add variants or use direct setters |
| Introspection disabled on endpoint | Medium | T4C | Graceful degradation with placeholder |
| Phase 3 layout algorithm | Low | T3 | Shallow trees; simple top-down suffices |
