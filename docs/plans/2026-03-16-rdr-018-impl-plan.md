# RDR-018 Implementation Plan: Query-Semantic LayoutStylesheet Extension

- **RDR**: 018
- **Status**: accepted
- **Date**: 2026-03-16
- **Epic**: Kramer-m2q

## Summary

Extend `LayoutStylesheet` to carry query-semantic and display properties per the SIEUFERD model (SIGMOD 2016). Three phases with independent deliverable timelines:

- **Phase 1** (implement now): Consolidate display properties under LayoutStylesheet infrastructure
- **Phase 2** (design only): Expression language sub-specification for client-side data manipulation
- **Phase 3** (deferred RDR): GraphQlUtil AST retention and query rewriting

## Key Findings from Research

- `LayoutStylesheet` interface is fully wired; `getBoolean`/`getVersion` exist (research-4)
- `hide-if-empty` and `sort-fields` are still read directly from `Relation` fields, NOT via stylesheet (research-5)
- `visible` property does not exist anywhere in the codebase (research-5)
- `LayoutPropertyKeys` constant class does not exist (research-5)
- `AutoLayout.decisionCache` does NOT key on stylesheet version — stale cache risk (research-4)
- `GraphQlUtil.buildSchema()` discards all AST structure — Phase 3 is a ground-up redesign (research-2, research-6)

### T1: LayoutPropertyKeys constant class (Kramer-f56)

**Unblocked.** Create `LayoutPropertyKeys` in `com.chiralbehaviors.layout.style`. All property key strings in one place, kebab-case convention. Initial keys:

```java
public final class LayoutPropertyKeys {
    public static final String VISIBLE       = "visible";
    public static final String RENDER_MODE   = "render-mode";
    public static final String HIDE_IF_EMPTY = "hide-if-empty";
    public static final String SORT_FIELDS   = "sort-fields";
}
```

Note: `sparkline` render-mode value deferred to RDR-019; add to this class when RDR-019 is accepted.

TDD: assert each constant's string value before implementation.

### T2: Wire hide-if-empty and sort-fields through LayoutStylesheet (Kramer-xu1)

**Depends on**: Kramer-f56

Migrate `RelationLayout.measure()` and `buildSortComparator()` to read `hide-if-empty` and `sort-fields` via `stylesheet.getBoolean(path, HIDE_IF_EMPTY, relation.isHideIfEmpty())` and `stylesheet.getString(path, SORT_FIELDS, relation.getSortFields())`. Relation field values become the defaults, preserving backward compatibility.

TDD: test that stylesheet overrides produce identical behavior to direct Relation field reads.

### T3: Add visible property to layout pipeline (Kramer-9cy)

**Depends on**: Kramer-f56

Add `visible` boolean property (default: `true`) per SchemaPath. When `visible=false`, the field is excluded from layout output entirely — not rendered, not measured. Evaluate at the top of `PrimitiveLayout.measure()` and `RelationLayout.measure()`.

TDD: test visible=false excludes field from output; test visible=true (default) is identity.

### T4: Wire stylesheet version into decisionCache key (Kramer-0cp)

**Depends on**: Kramer-9cy

`AutoLayout.decisionCache` must incorporate `LayoutStylesheet.getVersion()` in its cache key. Without this, `visible` and other stylesheet overrides silently produce stale layouts.

Source: `AutoLayout.java decisionCache`, `DefaultLayoutStylesheet.setOverride()` (increments version).

TDD: test that after a stylesheet override change, AutoLayout produces updated layout output.

## Phase 2: Data Manipulation Properties (DESIGN ONLY)

**Blocked on**: all Phase 1 tasks complete (Kramer-7x6 depends on Kramer-xu1, Kramer-9cy, Kramer-0cp)

### DESIGN Task: Expression language sub-specification (Kramer-7x6)

No implementation until this document exists. Must cover:

1. Grammar (EBNF) for filter predicates and formula expressions
2. Field reference scoping: sibling, parent, child aggregate resolution
3. Null handling: three-valued logic vs null-coalescing
4. Type coercion: JsonNode text/number/boolean for arithmetic/comparison
5. Virtual field materialization: formula results as JsonNode values
6. Formula-vs-aggregate distinction: per-row vs across-rows

Pipeline ordering (relational semantics): `filter → compute → aggregate → sort`

Deliverable: `docs/plans/2026-03-16-rdr-018-expr-lang-spec.md` or sub-RDR.

Properties for Phase 2 implementation (after spec):
- `filter` — string predicate expression
- `formula` — expression referencing sibling fields
- `aggregate` — enum: sum|avg|count|min|max|none
- `cell-format` — printf-style format pattern (moved from Phase 1; requires type coercion)

## Phase 3: Query Integration (DEFERRED RDR)

**Blocked on**: Phase 2 design complete (Kramer-e7y depends on Kramer-7x6)

### DESIGN Task: RDR for kramer-ql AST retention (Kramer-e7y)

A separate RDR is required before any implementation. Must address:

1. Redesign `GraphQlUtil.buildSchema()` to retain graphql-java `Document` AST alongside `SchemaNode` tree
2. Bidirectional mapping: SchemaPath ↔ query AST Field nodes
3. GraphQL query modification API: how stylesheet changes map to AST mutations
4. Cursor-based pagination per Relay spec (`first`/`after`, not LIMIT/OFFSET)
5. Backend-dependent properties (JOINON, COLUMNDEFINITION, REVERSEREF) — schema-dependent
6. FragmentSpread support (currently `UnsupportedOperationException`)

Phase 3 cannot begin until that RDR is accepted.
