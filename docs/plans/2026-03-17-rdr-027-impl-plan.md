# RDR-027 Implementation Plan: Interaction Model — Layer 7 Gestures

- **RDR**: 027
- **Date**: 2026-03-17
- **Status**: planned
- **Depends on**: RDR-026 (closed)

---

## Executive Summary

Implement the gesture layer (Layer 7) for Kramer's interaction model. A sealed `LayoutInteraction` hierarchy of 10 typed events drives mutations on `LayoutQueryState` (RDR-026) through an `InteractionHandler`. JavaFX context menus and column header click-to-sort provide the UI surface. Pagination-sort limitation is documented with a visual indicator.

Three phases: event model in `kramer` module, JavaFX bindings in `explorer` module, pagination awareness.

## Phase 1: Event Model + Handler (kramer module)

### Task 1a: fieldComparator descending sort prefix (Kramer-v4tp)

**No blockers — start immediately. Parallel with Task 1b.**

**Goal**: Enhance `RelationLayout.fieldComparator()` to support `-fieldName` prefix convention for descending sort.

**Context**:
- File: `kramer/src/main/java/com/chiralbehaviors/layout/RelationLayout.java` line 86
- Called from `buildSortComparator()` at lines 113, 115, and stylesheet-driven sort at lines 794, 796
- RDR-021 documents the `-` prefix convention for `sort-fields`

**Execution (TDD)**:

1. Create test class `kramer/src/test/java/com/chiralbehaviors/layout/FieldComparatorTest.java`
2. Test cases:
   - `fieldComparator("name")` sorts ascending (existing behavior)
   - `fieldComparator("-name")` sorts descending (reversed)
   - `fieldComparator("-age")` with numeric values sorts descending
   - Null/missing values still sort last regardless of direction
3. Modify `fieldComparator(String field)`:
   ```java
   static Comparator<JsonNode> fieldComparator(String field) {
       boolean descending = field.startsWith("-");
       String actualField = descending ? field.substring(1) : field;
       Comparator<JsonNode> cmp = (a, b) -> {
           // ... existing comparison logic using actualField ...
       };
       return descending ? cmp.reversed() : cmp;
   }
   ```
4. Verify: `mvn -pl kramer test -Dtest=FieldComparatorTest`
5. Run full module tests: `mvn -pl kramer test`

**Validation**:
- Descending comparator reverses order for both numeric and text values
- Null/missing values still sort last (reversed comparator must preserve this)
- All existing tests pass (no regression in buildSortComparator)

**Note on null handling**: `Comparator.reversed()` will reverse null-last to null-first. The implementation must handle this explicitly — either by wrapping nulls outside the reversed comparator or by using a two-stage approach (null check then reversed value comparison).

### Task 2b: Column header click-to-sort (Kramer-txh6)

**Blocked by**: Kramer-v4tp (Phase 1a), Kramer-x5l0 (Phase 1b)

**Goal**: Click-to-sort on ColumnHeader in table mode with 3-state toggle.

**Context**:
- RF-3: ColumnHeader has zero interaction currently
- File: `kramer/src/main/java/com/chiralbehaviors/layout/table/ColumnHeader.java`
- Handler attached via lambda capture closing over SchemaPath at construction time
- Sort state machine: none -> asc -> desc -> clear (read from `queryState.getFieldState(path).sortFields()`)

**Execution**:

1. Write test for sort state toggle cycle:
   - Initial state: no sort -> click -> sortFields == "leaf"
   - Click again -> sortFields == "-leaf"
   - Click again -> sortFields == null

2. Add constructor parameter or setter to `ColumnHeader` for sort click callback:
   ```java
   public ColumnHeader(double width, double height, PrimitiveLayout layout,
                       Runnable onSortClick) {
       this(width, height, layout);
       setOnMouseClicked(e -> {
           if (e.getButton() == MouseButton.PRIMARY) onSortClick.run();
       });
   }
   ```
   Alternative: attach handler in `RelationLayout.buildColumnHeader()` or in the explorer wiring layer.

3. The sort toggle logic lives in the gesture wiring (Task 2c) — ColumnHeader just provides the click surface.

4. Verify: `mvn -pl kramer test`

**Design note**: ColumnHeader is in the `kramer` module which has no dependency on `InteractionHandler`. The handler lambda is injected from the explorer module at wiring time. ColumnHeader only needs to accept and invoke a `Runnable` or `EventHandler<MouseEvent>`.

## Risk Register

| Risk | Severity | Mitigation |
|------|----------|------------|
| Null handling in reversed fieldComparator | Medium | Explicit null-last logic outside reversed() |
| Context menus on recycled VirtualFlow cells | Low | Attach at bind time, detach at unbind |
| Name collision QueryState vs LayoutQueryState | Low | Documented, use FQN |
| Rapid sort toggles queue multiple layout passes | Low | Acceptable per RF-5, no debouncing in scope |
| ColumnHeader constructor signature change | Low | Overloaded constructor preserves backward compat |

## Success Criteria (from RDR-027)

- [ ] All 10 interaction events in the sealed hierarchy are handled by InteractionHandler
- [ ] Column header click sorts ascending, click again sorts descending, click again clears sort
- [ ] Right-click context menu on any field offers Hide, Sort, Filter (where applicable)
- [ ] Right-click on Relation offers render mode choices (table, outline, sparkline where applicable)
- [ ] ResetAll clears all QueryState overrides and re-layouts to default
- [ ] Pagination-sort limitation is visually indicated when applicable
- [ ] All gestures work with both outline and table rendering modes
- [ ] Zero changes to AutoLayout, RelationLayout, or PrimitiveLayout core (except fieldComparator enhancement)
