## Executive Summary

Extend the constraint solver from binary (TABLE/OUTLINE) to ternary (TABLE/OUTLINE/CROSSTAB) for CROSSTAB-eligible Relation nodes. The search space becomes `3^K x 2^(N-K)` where K is the number of eligible nodes. This gives the solver global awareness of CROSSTAB width contributions, enabling adaptive mode selection based on available space rather than the current all-or-nothing hard override.

Three sequential tasks implement the RDR specification:
1. **T1** — Extend `RelationConstraint` with CROSSTAB feasibility data
2. **T2** — Extend `ExhaustiveConstraintSolver` for ternary decisions + rendering wiring
3. **T3** — Integration tests with mixed TABLE/OUTLINE/CROSSTAB schemas

### CROSSTAB Width Formula

```
crosstabWidth = rowHeaderWidth + pivotCount * pivotColumnWidth + insets

Where:
  rowHeaderWidth   = sum of non-pivot, non-value child column widths
  pivotCount       = measureResult.pivotStats().pivotCount()
  pivotColumnWidth = value field's MeasureResult.columnWidth()
  insets           = style.getNestedHorizontalInset()
```

## Phase 4a-T2: Extend ExhaustiveConstraintSolver for Ternary Decisions

**Bead**: Kramer-lwmu
**Depends on**: Kramer-fr3i (T1)
**Estimated scope**: Medium-large (solver algorithm + rendering wiring)

### Files to Modify

| File | Change |
|------|--------|
| `kramer/src/main/java/.../ExhaustiveConstraintSolver.java` | Replace bit-vector with mixed-radix enumeration; extend `checkFeasible()` for CROSSTAB; new objective scoring; greedy fallback extension |
| `kramer/src/main/java/.../RelationLayout.java` | Wire solver CROSSTAB assignment to `layoutCrosstab()` in `layout()` (RF-5 fix) |
| `kramer/src/test/.../ConstraintSolverTest.java` | Add ternary solver unit tests |

### Execution Steps (TDD)

1. Write new tests in `ConstraintSolverTest.java`:
   - **Single eligible node**: CROSSTAB when fits, TABLE when CROSSTAB does not, OUTLINE when neither
   - **Mixed schema**: 2 eligible + 3 non-eligible nodes, verify all `3^2 x 2^3 = 72` assignments covered
   - **Performance**: N=15 with K=5, search space ~249K assignments, must complete in <1ms with pruning
   - **Greedy fallback with eligible nodes**: verify CROSSTAB-first preference ordering
   - **Objective scoring**: verify `score = 2*(tableCount + crosstabCount) + tableCount` ranking
2. Implement mixed-radix enumeration in `ExhaustiveConstraintSolver`:
   - Per-node radix array: `radix[i] = crosstabEligible ? 3 : 2`
   - Total assignments: `product(radix[i])` instead of `1 << n`
   - Mode mapping: `0 = OUTLINE, 1 = TABLE, 2 = CROSSTAB`
   - DFS-order variable list preserved for pruning invariant
3. Extend `checkFeasible()`:
   - Add CROSSTAB branch: `crosstabWidth <= availableWidth`
   - CROSSTAB parent: children see unbounded width (same as OUTLINE parent, per Phase 1 approximation)
4. Implement new objective scoring:
   - `score = 2 * (tableCount + crosstabCount) + tableCount`
   - Primary term: column-oriented (TABLE or CROSSTAB) > OUTLINE
   - Secondary term: TABLE preferred over CROSSTAB as tie-break
5. Extend greedy fallback:
   - For eligible nodes: try CROSSTAB first (if feasible), then TABLE, then OUTLINE
   - Matches solver objective ordering
6. **Decision required**: mixed-radix threshold strategy. Choose one:
   - Option A: `effectiveThreshold = 15 - K` (conservative)
   - Option B: Keep threshold at 15, add performance test for K=5 N=15 (<1ms)
   - Document the decision in a code comment
7. Wire solver CROSSTAB assignment to rendering (RF-5 fix):
   - In `RelationLayout.layout()`: when `solverResults` contains CROSSTAB for this node, call `layoutCrosstab(width, pivotValues)` instead of `columnWidth()`
   - Verify: solver assigns CROSSTAB -> rendered control is `CrosstabHeader`, not outline column

### Acceptance Criteria

- [ ] Mixed-radix enumeration covers all `3^K x 2^(N-K)` assignments for mixed schemas
- [ ] Single eligible node: CROSSTAB when fits, TABLE when not, OUTLINE when neither
- [ ] Objective scoring ranks TABLE/CROSSTAB above OUTLINE, TABLE above CROSSTAB for tie-break
- [ ] Greedy fallback produces consistent results with solver preference ordering
- [ ] N=15 K=5 performance: <1ms with pruning
- [ ] `checkFeasible()` prunes infeasible CROSSTAB assignments

- [ ] Mixed-radix threshold decision documented in code
- [ ] Code compiles including all tests

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Search space growth for large K | Medium | K is always small (pivot-field is intentional config); N>15 greedy fallback retained |
| CROSSTAB width is data-dependent | Medium | Re-measure regenerates RelationConstraint records; no new invalidation needed |
| Mixed-radix breaks pruning invariant | Low | Pruning walks constraint tree, not enumeration counter; DFS order preserved |
| CROSSTAB vs TABLE preference ambiguity for small pivot counts | Low | Default TABLE tie-break; `prefer-mode` property deferred |

## Rollback Strategy

`crosstabEligible` defaults to `false` for all existing nodes. If mixed-radix enumeration introduces regressions, revert `ExhaustiveConstraintSolver` to bit-vector — `crosstabEligible == false` for all nodes means mixed-radix degenerates to the Phase 1 bit-vector case. Rollback is a single-class revert.
