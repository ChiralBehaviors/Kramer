# RDR-012 Implementation Plan: Reactive Semantic Constraint Layout

**Date**: 2026-03-16
**RDR**: docs/rdr/RDR-012-reactive-semantic-constraint-layout.md
**Status**: accepted (2026-03-16)

---

## Executive Summary

Replace the single greedy `if`-statement at `RelationLayout.layout():613` with a globally-optimal exhaustive constraint solver for binary TABLE/OUTLINE mode assignment across all Relation nodes in a schema tree. This is Phase 1 of a four-phase research effort. Phases 2–4 are design beads (future sprints) that build additively on the Phase 1 constraint model.

The change is near-surgical: one call-site replaced, new infrastructure additive, rollback is revert of that one line.

---

## Phase 1 — Implement Now (near-surgical)

### T1: ConstraintSolver interface + ExhaustiveConstraintSolver
**Bead**: Kramer-044
**Priority**: P2
**Unblocked**: yes (no dependencies)

**What**: Define `ConstraintSolver` interface and `ExhaustiveConstraintSolver`.

Key types:
- `RelationConstraint` record: `(SchemaPath path, double tableWidth, double nestedHorizontalInset, double outlineWidth, boolean hardOverride, RelationRenderMode forcedMode)`
- `ConstraintSolver` interface: `Map<SchemaPath, RelationRenderMode> solve(List<RelationConstraint> constraints, double availableWidth)`
- `ExhaustiveConstraintSolver`: enumerate 2^N bitsets over non-hardOverride nodes; score = sum of (depth * tableBonus) for TABLE assignments; return best feasible assignment; N>15 falls back to greedy with `LOG.warn`.

**Objective**: maximize TABLE assignments; depth-first (deeper TABLE = higher score for tiebreak).

**Feasibility test per assignment**: for each TABLE node, `tableWidth + nestedHorizontalInset <= width`. All TABLE assignments in the bitset must be feasible for the assignment to be accepted.

**N>15 fallback**: replicate current `if (tableWidth + style.getNestedHorizontalInset() <= width)` greedy per-node logic.

**Package**: `com.chiralbehaviors.layout` (alongside `LayoutDecisionKey`, `SchemaPath`).

**TDD test class**: `ConstraintSolverTest` — write before implementation:
1. Single-node: TABLE when fits, OUTLINE when not
2. N=2: solver picks globally better assignment than greedy would choose independently
3. N=15: completes in <1ms (timing assertion)
4. N=16: triggers fallback, diagnostic logged
5. hardOverride=true node: excluded from enumeration, assigned forcedMode

### T2: Wire ExhaustiveConstraintSolver into RelationLayout.layout()
**Bead**: Kramer-7pp
**Priority**: P2
**Depends on**: Kramer-044

**What**: Replace the greedy if-statement at `RelationLayout.layout():613` with solver invocation.

Current code (line 613):
```java
if (tableWidth + style.getNestedHorizontalInset() <= width) {
    return nestTableColumn(Indent.TOP, new Insets(0));
}
```

Replacement strategy:
- Solver is called **once at the layout root** (not per-node) after the children measurement pre-pass.
- A pre-pass walk collects `RelationConstraint` for every Relation node in the subtree.
- LayoutStylesheet `RENDER_MODE` overrides → `hardOverride=true, forcedMode=<mode>`.
- CROSSTAB nodes (isCrosstab=true) → `hardOverride=true, forcedMode=CROSSTAB` (Phase 4a domain).
- Each `RelationLayout` reads its assigned mode from the solver result map keyed by `getSchemaPath()`.

**TDD test class**: `RelationLayoutConstraintWiringTest` — write before modifying `RelationLayout`:
1. All existing `RelationComputeTest` cases pass (greedy parity regression)
2. Solver-chosen TABLE on a case greedy would choose OUTLINE (must use the improvement case from T1 tests)
3. CROSSTAB node: solver skips it, isCrosstab path unchanged

### T3: Integration tests
**Bead**: Kramer-m1i
**Priority**: P2
**Depends on**: Kramer-7pp

**What**: Integration tests validating Phase 1 success criteria.

**Test class**: `ConstraintSolverIntegrationTest`

Required test cases:
1. **Greedy parity**: all existing test schemas produce identical TABLE/OUTLINE assignments.
2. **Improvement case A**: a 2-level schema where the root greedily chooses OUTLINE (table width slightly exceeds root width), but the solver finds a child-TABLE assignment that is globally better. Assert more content visible.
3. **Improvement case B**: a 3-level schema with crossed dependencies. Document the schema in a comment explaining exactly why greedy fails.
4. **N>15 fallback**: 16-node schema triggers fallback; diagnostic in log.
5. **Hard override**: stylesheet forces OUTLINE on a node that would otherwise be TABLE; solver respects it.
6. **CROSSTAB bypass**: node with `isCrosstab=true` is excluded from solver search space.

Success criteria (from RDR-012):
- [ ] Constraint solver is never worse than greedy on any existing test
- [ ] At least 2 documented improvement cases with measurable density gain

## Phase 2 — Future Sprint (LLM Annotation Pipeline)

**Bead**: Kramer-d70
**Priority**: P3
**Depends on**: Kramer-044 (Phase 1 constraint model)

Design bead only. Implements RDR-012 Component 2: LLM schema annotation for semantic density weighting.

Prerequisite gate before starting: select and validate LP solver (Apache Commons Math SimplexSolver or ojAlgo). ojAlgo is the named candidate (pure Java, Apache 2.0).

Key scope: annotation schema, prompt template, `AnnotatedSchema`, soft constraint cost functions with `@importance` multipliers (high=10, medium=5, low=1 — to be calibrated), `@`-prefix namespace for LayoutPropertyKeys.

Coordination: RDR-018 closed; implement against existing `LayoutPropertyKeys`.

## Phase 3 — Future Sprint (Reactive Invalidation)

**Bead**: Kramer-5gv
**Priority**: P3
**Depends on**: Kramer-044 (Phase 1 constraint model)

Design bead only. Implements RDR-012 Component 3: schema tree as streaming invalidation graph.

Gap to close before starting: `frozenResult` clears on stylesheet version change only, not data change.

Key scope: `ObservableData` adapter, path-level diff emission, MeasureResult invalidation listeners, partial re-solve for affected subtrees.

Three complexity regimes to handle explicitly:
- (a) O(log N) point updates within existing statistical range
- (b) O(items × depth) max-value removal/insertion (requires full rescan)
- (c) Outline sibling-lateral: parent-scoped invalidation

Target: <16ms relayout for 20-node schema on case (a) updates.

Coordination: RDR-016 closed; Phase 3 enhances the live LayoutDecisionKey cache.

## Rollback Strategy

- `ConstraintSolver` infrastructure remains inert in codebase.
- No other code paths affected.
