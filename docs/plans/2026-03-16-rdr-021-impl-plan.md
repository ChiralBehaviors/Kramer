# RDR-021 Implementation Plan: Expression Language for LayoutStylesheet

- **RDR**: 021
- **Date**: 2026-03-16
- **Design spec**: docs/plans/2026-03-16-rdr-018-expression-language-spec.md
- **Status**: planned

---

## Executive Summary

Implement a zero-dependency recursive descent expression language embedded in `LayoutStylesheet` string property values. Enables row filtering, formula columns, group-level aggregation, and derived sort keys within the layout pipeline. Java package: `com.chiralbehaviors.layout.expression`.

Three strictly sequenced steps: parser (Step A) → evaluator (Step B) → pipeline wiring (Step C).

## Phase Breakdown

### Step A: Lexer and Recursive Descent Parser (Kramer-hqtt)

**No blockers — start immediately.**

**Goal**: Produce the sealed `Expr` AST hierarchy and a parser that turns expression strings into AST.

**New Java package**: `com.chiralbehaviors.layout.expression`

**Files to create** (TDD order):

1. `ExpressionParserTest.java` (test module, under `kramer`) — write first
2. `Expr.java` — sealed interface + record implementations
3. `Token.java` — token types + record
4. `Lexer.java` — single-pass tokenizer
5. `Parser.java` — recursive descent parser
6. `ParseException.java` — checked exception with char offset

**AST node types**:

```java
sealed interface Expr permits Literal, FieldRef, BinaryOp, UnaryOp, FunctionCall, AggregateCall {}

record Literal(Object value) implements Expr {}               // Double, String, Boolean, or null
record FieldRef(List<String> path) implements Expr {}         // $name or ${a.b.c}
record BinaryOp(BinaryOp.Op op, Expr left, Expr right) implements Expr {}
record UnaryOp(UnaryOp.Op op, Expr operand) implements Expr {}
record FunctionCall(String name, List<Expr> args) implements Expr {}
record AggregateCall(String fn, Optional<Expr> arg) implements Expr {}  // count() has empty arg
```

**Token set**: `NUMBER, STRING, BOOLEAN, NULL, IDENT, FIELD_REF, FIELD_PATH_REF, PLUS, MINUS, STAR, SLASH, BANG, AMP_AMP, PIPE_PIPE, EQ, NEQ, LT, GT, LTE, GTE, LPAREN, RPAREN, COMMA, EOF`

**Operator precedence** (highest to lowest): unary `!`/`-`, `*`/`/`, `+`/`-`, comparisons, `&&`, `||`

**Parse-time validations** (all log WARN + treat expression as absent):
- Aggregate function in non-aggregate-expression context
- Mixed aggregate and non-aggregate at same expression root (e.g., `sum($x) + $y`)
- Expression length > 4096 characters (security cap)

**`AggregateCall` vs `FunctionCall`**: distinguished at parse time by checking the function name against the aggregate set (`sum`, `count`, `avg`, `min`, `max`). `count()` → `AggregateCall("count", Optional.empty())`.

**`ParseException`**: checked exception, carries `int offset` (char position in input string) and message.

**Acceptance criteria**:
- All grammar productions from spec Section 1 are covered
- `Parser.parse(String)` returns correct AST for all example expressions in spec Sections 4.1–4.4
- Malformed input produces `ParseException` with non-negative char offset
- Zero new compile-time dependencies

**Test cases to cover**:
- Arithmetic: `$price * $quantity`
- String concat: `upper($last_name) + ", " + $first_name`
- Comparison: `$revenue > 0 && $status != "cancelled"`
- Nested field: `${customer.tier} == "premium"`
- Aggregate: `sum($revenue)`, `count()`, `avg($unit_price)`
- `if`: `if($discount_rate > 0, $price * (1 - $discount_rate), $price)`
- Null literal: `$field == null`
- Error: unterminated string → ParseException at correct offset
- Error: `sum($x) + $y` at aggregate-expression → WARN + absent

### Step B: Pattern-Matching Evaluator (Kramer-267n)

**Blocked on Kramer-hqtt (Step A).**

**Goal**: Tree-walking evaluator over the sealed `Expr` hierarchy. Stateless per expression; cached ASTs.

**New files**:

1. `ExpressionEvaluatorTest.java` (test module) — write first
2. `ExpressionEvaluator.java` — main evaluator class
3. `EvalContext.java` — immutable per-row evaluation context (wraps `JsonNode` row)

**Key methods**:

```java
// Per-row: filter, formula, sort-expression
Object evaluate(Expr expr, JsonNode row);

// Aggregate: cross-row reduction
Object evaluateAggregate(AggregateCall call, List<JsonNode> rows);

// Cache management
Expr compile(String expression);          // parse + cache by string
void invalidateCache(int stylesheetVersion);
```

**Evaluator dispatch** (Java 25 pattern matching):

```java
Object evaluate(Expr expr, JsonNode row) {
    return switch (expr) {
        case Literal(var v)                    -> v;
        case FieldRef(var path)                -> resolveField(row, path);
        case BinaryOp(var op, var l, var r)    -> evalBinary(op, evaluate(l, row), evaluate(r, row));
        case UnaryOp(var op, var e)            -> evalUnary(op, evaluate(e, row));
        case FunctionCall(var fn, var args)    -> evalFunction(fn, args, row);
        case AggregateCall(_, _)               -> throw new EvalException("aggregate in row context");
    };
}
```

**Short-circuit evaluation**:
- `&&`: if left evaluates to false (or null→false), right is not evaluated
- `||`: if left evaluates to true, right is not evaluated

**Type materialization** (JsonNode → Object):

| JsonNode kind | Java type |
|---------------|-----------|
| IntNode, LongNode, DoubleNode, FloatNode, DecimalNode | Double |
| TextNode | String |
| BooleanNode | Boolean |
| NullNode, MissingNode, ArrayNode, ObjectNode | null |

**Null propagation**: arithmetic with null → null; `&&`/`||` null → false; `!null` → true; division by zero → null.

**AST cache**: `Map<String, Expr>` keyed by expression string. Invalidated when `LayoutStylesheet.getVersion()` changes. Thread-safety: cache is per-evaluator instance; evaluator is accessed from JavaFX Application Thread only.

**Cycle detection** (for formula dependency graphs):
- Input: `Map<String, Set<String>>` of field → fields-it-references
- Output: set of fields participating in cycles
- Algorithm: Kahn's topological sort; fields in cycles are removed and logged at WARN
- Result cached per stylesheet version alongside AST cache

**Wiring-time validations** (once per expression per stylesheet version):
- `aggregate-expression` AST has no `AggregateCall` → WARN + treat as absent
- `formula-expression` AST has an `AggregateCall` → WARN + treat as absent

**Acceptance criteria**:
- All spec Section 2 null propagation table entries covered by tests
- Aggregate reduction tests: sum/count/avg/min/max with null exclusion
- Cycle detection test: A→B→A cycle produces WARN, both fields treated as absent
- Self-reference (A→A) detected by same cycle algorithm
- Cache invalidation test: new stylesheet version clears cached AST

### Step C: Pipeline Wiring into RelationLayout.measure() (Kramer-c43t)

**Blocked on Kramer-267n (Step B).**

**Goal**: Wire `ExpressionEvaluator` into the `RelationLayout.measure()` data pipeline. Add 4 property keys. Resolve 4 open questions. Integration tests.

**Files to modify**:

1. `LayoutPropertyKeysTest.java` (test module) — verify constant values first
2. `LayoutPropertyKeys.java` — add 4 new constants
3. `Style.java` — inject `ExpressionEvaluator`
4. `RelationLayout.java` — add pipeline stages

**New LayoutPropertyKeys constants**:

```java
public static final String FILTER_EXPRESSION    = "filter-expression";
public static final String FORMULA_EXPRESSION   = "formula-expression";
public static final String AGGREGATE_EXPRESSION = "aggregate-expression";
public static final String SORT_EXPRESSION      = "sort-expression";
```

**`Style` factory injection**: `Style` already receives `LayoutStylesheet`. Add `ExpressionEvaluator` as a field initialized from the stylesheet (or a factory method). Pass to `RelationLayout` at construction.

**Pipeline stages in `RelationLayout.measure()`** — insert after hideIfEmpty filter (~line 720):

```
// 1. filter-expression (after hideIfEmpty, before child measure)
if (stylesheet != null && myPath != null) {
    String filterExpr = stylesheet.getString(myPath, FILTER_EXPRESSION, null);
    if (filterExpr != null) {
        Expr filterAst = evaluator.compile(filterExpr);  // ParseException → absent
        datum = filterArrayNode(datum, row -> Boolean.TRUE.equals(toBoolean(evaluator.evaluate(filterAst, row))));
    }
}

// 2. formula-expression (for each child Primitive)
//    build dependency graph, detect cycles, evaluate in topological order
//    materialize results as ObjectNode overlays on each row

// 3. aggregate-expression (for aggregate-marked Primitives)
//    evaluate over post-formula row list; hold result for render phase

// 4. sort-expression (per row, before final sort)
//    if sort-expression present: evaluate per row, use result as sort key
//    sort-expression takes precedence over sort-fields; sort-fields is tiebreaker
```

**Formula injection strategy (Open Question 2)**: create a shallow `ObjectNode` overlay per row containing formula results merged with the original fields. Use `JsonNodeFactory.instance.objectNode()` and `setAll(originalObjectNode)` + `set(fieldName, formulaResult)`. Avoids deepCopy; overlay wraps the original node. Memory cost: O(rows × formula-fields) ObjectNode references, not full copies.

**`cell-format` interaction (Open Question 1)**: catch `IllegalFormatException` when formatting formula results; log WARN; render raw value as fallback.

**Aggregate render position (Open Question 3)**: `aggregate-expression` computation is delivered in Step C. Rendering is deferred to a follow-on task once `aggregate-position` property is defined. In Step C, aggregate results are stored in `MeasureResult` (new field) for later use.

**Sort precedence (Open Question 4)**: document in `LayoutPropertyKeys` javadoc on `SORT_EXPRESSION`: "When both sort-expression and sort-fields are present, sort-expression keys take precedence; sort-fields serves as a tiebreaker."

**RDR-013 ordering validation**: confirm that the formula pass completes before `RelationLayout.measure()` calls child measure (which triggers RDR-013 width sampling). The pipeline position above satisfies this if RDR-013 sampling is inside the child measure path — validate by inspection during implementation.

**RDR-016 compatibility**: review `MeasureResult` structure for any changes from RDR-016 before adding aggregate result fields.

**Integration test scenarios**:

1. Filter: `$revenue > 0` excludes rows with revenue ≤ 0; verify row count in MeasureResult
2. Formula: `$price * $quantity` produces correct per-row total
3. Aggregate: `sum($revenue)` produces correct sum across filtered rows
4. Sort: `lower($last_name)` sorts case-insensitively
5. Combined: filter + formula + sort in single measure pass
6. Cache: identical measure at same width does not re-invoke evaluator
7. Stylesheet version change: cache cleared, expressions re-compiled on next measure

**Acceptance criteria**:
- All existing tests pass
- No `ScriptEngine`, Nashorn, GraalVM, or reflection-based evaluation paths
- Expression parse errors logged at WARN, pipeline stage skipped, layout continues
- 4 open questions documented in javadoc or follow-on RDR

## Critical Path

The single critical path is A → B → C. No tasks can be parallelized within this feature.

**Risk items**:
- Formula `JsonNode` injection strategy (Open Question 2) — resolved in Step C to shallow overlay, avoiding deepCopy
- RDR-016 `MeasureResult` structural changes — must be reviewed before Step C adds aggregate result fields
- RDR-013 ordering — must be validated by inspection during Step C; if sampling hook is not in child measure path, move formula pass earlier
