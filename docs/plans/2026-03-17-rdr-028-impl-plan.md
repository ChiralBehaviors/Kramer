# RDR-028 Implementation Plan: Query Rewriter -- Layer 1<->2 Bridge

- **RDR**: 028
- **Date**: 2026-03-17
- **Status**: planned
- **Depends on**: RDR-020 (closed), RDR-026 (closed), RDR-027 (closed)

---

## Executive Summary

Implement the Query Rewriter that bridges Layer 2 (QueryState) to Layer 1 (GraphQL query), pushing filter, sort, and visibility operations to the server when the backend supports them. The system discovers backend capabilities via GraphQL schema introspection, rewrites queries with server-side arguments, and falls back to client-side evaluation (RDR-021 pipeline) when push-down is not available.

Five phases across two modules (kramer-ql, explorer), with Phase 3a parallelizable with Phases 1 and 2.

## Phase 1: ServerCapabilities Discovery (Kramer-ycbd)

**Module**: kramer-ql
**Dependencies**: None (entry point)

### Goal

Introspect a GraphQL endpoint's schema to discover which Relation fields accept filter, sort, limit, and aggregate arguments. Produce an immutable `ServerCapabilities` record.

### New Files

| File | Purpose |
|------|---------|
| `kramer-ql/src/main/java/.../graphql/ServerCapabilities.java` | Immutable record: `PushableOperation` enum + `Map<SchemaPath, Set<PushableOperation>>` |
| `kramer-ql/src/main/java/.../graphql/SchemaIntrospector.java` | Executes introspection query, parses `__schema` result, maps argument names to operations |
| `kramer-ql/src/test/java/.../graphql/ServerCapabilitiesTest.java` | Unit tests against fixture JSON |
| `kramer-ql/src/test/resources/.../introspection-relay.json` | Relay-style schema fixture (where, orderBy) |
| `kramer-ql/src/test/resources/.../introspection-hasura.json` | Hasura-style schema fixture (where, order_by) |
| `kramer-ql/src/test/resources/.../introspection-bare.json` | Minimal schema with no filter/sort arguments |

### Design

```java
public record ServerCapabilities(
    Map<SchemaPath, Set<PushableOperation>> fieldCapabilities
) {
    public enum PushableOperation {
        FILTER, SORT, LIMIT, AGGREGATE
    }

    public boolean supports(SchemaPath path, PushableOperation op) {
        return fieldCapabilities.getOrDefault(path, Set.of()).contains(op);
    }

    /** No capabilities -- pure client-side mode. */
    public static final ServerCapabilities NONE =
        new ServerCapabilities(Map.of());
}
```

`SchemaIntrospector`:
- `discover(WebTarget endpoint)` -- executes `{ __schema { types { ... } } }` introspection query
- `fromIntrospectionResult(JsonNode result, String queryString)` -- pure parser, unit-testable
- Recognized argument name patterns:
  - FILTER: `filter`, `where`
  - SORT: `orderBy`, `order_by`, `sort`
  - LIMIT: `first`, `limit`, `take`
  - AGGREGATE: `aggregate`, `_aggregate`

### Test Strategy

- Parse each fixture JSON, assert correct PushableOperation mapping per SchemaPath
- Test `ServerCapabilities.NONE` fallback
- Test introspection parse error graceful fallback (returns NONE)
- Test fields with no arguments produce empty capability set
- Test fields with only `id` argument produce empty capability set

### Execution Instructions

1. Write `ServerCapabilitiesTest.java` first (TDD)
2. Create fixture JSON files
3. Implement `ServerCapabilities` record
4. Implement `SchemaIntrospector`
5. Verify all tests pass, run `mvn -pl kramer-ql test`

## Phase 2: QueryRewriter Core (Kramer-g9ox)

**Module**: kramer-ql (primary), kramer (minor accessor addition)
**Dependencies**: Phase 1 (Kramer-ycbd)

### Goal

Build the `QueryRewriter` that takes a `SchemaContext`, `LayoutQueryState`, and `ServerCapabilities`, and produces a rewritten GraphQL query string with server-side filter/sort arguments injected.

### New Files

| File | Purpose |
|------|---------|
| `kramer-ql/src/main/java/.../graphql/QueryRewriter.java` | Core rewriter: checks capabilities, patches Document AST, delegates to QueryBuilder |
| `kramer-ql/src/main/java/.../graphql/ExprToGraphQL.java` | Translates `Expr` AST nodes to GraphQL argument values |
| `kramer-ql/src/test/java/.../graphql/QueryRewriterTest.java` | Unit tests for rewriting logic |
| `kramer-ql/src/test/java/.../graphql/ExprToGraphQLTest.java` | Unit tests for expression translation |

### Modified Files

| File | Change |
|------|--------|
| `kramer/src/main/java/.../query/LayoutQueryState.java` | Add `Set<SchemaPath> overriddenPaths()` accessor returning `fieldStates.keySet()` copy |

### Design

```java
public class QueryRewriter {
    private final ServerCapabilities capabilities;

    public QueryRewriter(ServerCapabilities capabilities) {
        this.capabilities = capabilities;
    }

    public String rewrite(SchemaContext ctx, LayoutQueryState queryState) {
        // 1. For each path with filter/sort expression in queryState:
        //    a. Check capabilities.supports(path, FILTER/SORT)
        //    b. If pushable: parse expression, translate via ExprToGraphQL,
        //       add argument to Field in Document AST
        // 2. Visible=false already handled by QueryBuilder.reconstruct()
        // 3. Delegate to QueryBuilder.reconstruct(modifiedCtx, queryState)
        // Returns the final query string
    }
}
```

`ExprToGraphQL` translation rules (v1, fixed-format):

| Expr node | GraphQL argument format |
|-----------|------------------------|
| `BinaryOp(EQ, FieldRef, Literal)` | `{field: {_eq: value}}` |
| `BinaryOp(NEQ, FieldRef, Literal)` | `{field: {_neq: value}}` |
| `BinaryOp(LT/GT/LTE/GTE, ...)` | `{field: {_lt/_gt/_lte/_gte: value}}` |
| `BinaryOp(AND, left, right)` | `{_and: [left, right]}` |
| `BinaryOp(OR, left, right)` | `{_or: [left, right]}` |
| Sort: `"-name"` | `{name: desc}` |
| Sort: `"name"` or `"+name"` | `{name: asc}` |
| `ScalarCall`, `AggregateCall` | Not pushable (return null) |
| Arithmetic `ADD/SUB/MUL/DIV` | Not pushable (return null) |

### Test Strategy

- **QueryRewriterTest**:
  - Test with all-pushable ServerCapabilities: verify filter/sort arguments appear in rewritten query
  - Test with no-pushable: verify original query string returned unchanged
  - Test mixed: some fields pushable, some not
  - Test visible=false field removal (via QueryBuilder)
  - Parse rewritten query string back to Document, verify argument presence
- **ExprToGraphQLTest**:
  - Test each BinaryOp translation
  - Test AND/OR nesting
  - Test non-pushable expressions return null
  - Test sort field parsing ("-name", "name", "+name")

### Execution Instructions

1. Add `overriddenPaths()` to `LayoutQueryState` (1 line)
2. Write `ExprToGraphQLTest.java` first (TDD)
3. Implement `ExprToGraphQL`
4. Write `QueryRewriterTest.java` (TDD)
5. Implement `QueryRewriter`
6. Verify: `mvn -pl kramer test` and `mvn -pl kramer-ql test`

## Phase 3b: Change-Driven Re-fetch Path (Kramer-tnya)

**Module**: explorer
**Dependencies**: Phase 2 (Kramer-g9ox), Phase 3a (Kramer-vhzt)

### Goal

Wire the QueryRewriter into AutoLayoutController's event loop. When LayoutQueryState changes and a rewriter is available, rewrite the query and re-fetch from the server instead of just re-laying out client-side data.

### Modified Files

| File | Change |
|------|--------|
| `explorer/src/main/java/.../explorer/AutoLayoutController.java` | Add `queryRewriter`, `serverCapabilities`, `fetchInProgress` fields; replace change listener; update `ActiveState.fetchAsync()` to clear guard; wire capabilities on endpoint change |

### Design

New fields in `AutoLayoutController`:

```java
private QueryRewriter queryRewriter;
private ServerCapabilities serverCapabilities;
private volatile boolean fetchInProgress;
```

Replace the change listener registered in `initialize()` at line 274:

```java
// Current: layoutQueryState.addChangeListener(() -> layout.autoLayout());
// Replace with:
layoutQueryState.addChangeListener(() -> {
    if (fetchInProgress) return;
    if (queryRewriter != null && schemaContext != null) {
        fetchInProgress = true;
        String rewritten = queryRewriter.rewrite(schemaContext, layoutQueryState);
        activeQuery.fetchAsync(rewritten);
    } else {
        layout.autoLayout();
    }
});
```

Update `ActiveState.fetchAsync()` to clear the guard:

```java
public void fetchAsync(String query) {
    executor.submit(() -> {
        String result;
        try {
            result = GraphQlUtil.evaluate(endpoint, query);
        } catch (Exception e) {
            log.error("GraphQL fetch failed", e);
            result = "{}";
        }
        final String finalResult = result;
        Platform.runLater(() -> {
            setData(finalResult);
            JSObject window = (JSObject) webEngine.executeScript("window");
            window.call("onFetchComplete", finalResult);
            fetchInProgress = false; // AFTER setData to prevent re-entry
        });
    });
}
```

Wire ServerCapabilities on endpoint change (in `ActiveState.setTargetURL()`):

```java
@Override
public void setTargetURL(String targetURL) {
    // ... existing URL validation ...
    endpoint = ClientBuilder.newClient().target(uri);
    // Discover capabilities asynchronously
    executor.submit(() -> {
        serverCapabilities = SchemaIntrospector.discover(endpoint);
        queryRewriter = new QueryRewriter(serverCapabilities);
    });
}
```

User-typed queries from the WebEngine should bypass the rewriter. The existing `fetchAsync()` path from the WebEngine already takes a raw query string -- the rewriter only fires from the LayoutQueryState change listener, not from user query submission.

  - Verify rewritten query is received by stub server

- **Guard test**: apply two rapid LayoutQueryState changes, verify only one fetch occurs
- **Fallback test**: set serverCapabilities=null, verify autoLayout() fires instead

## Phase 3c: Double-evaluation Prevention + End-to-end Test (Kramer-bnl6)

**Module**: kramer-ql (modify QueryRewriter), explorer (E2E test)
**Dependencies**: Phase 3b (Kramer-tnya)

### Goal

When the QueryRewriter pushes an operation server-side, null the corresponding client-side expression on LayoutQueryState to prevent double evaluation. Retain original expressions internally for re-push on the next rewrite cycle.

### Modified Files

| File | Change |
|------|--------|
| `kramer-ql/src/main/java/.../graphql/QueryRewriter.java` | Add `pushedExpressions` map; null client-side expressions inside `suppressNotifications`; detect user clearing (null) to stop pushing |

### New Files

| File | Purpose |
|------|---------|
| `kramer-ql/src/test/java/.../graphql/QueryRewriterDoubleEvalTest.java` | Unit test: verify nulling behavior and expression retention |
| `explorer/src/test/java/.../explorer/EndToEndRewriteTest.java` | Full integration: interaction -> QueryState -> rewrite -> fetch -> layout |

### Design

Add internal state to `QueryRewriter`:

```java
public class QueryRewriter {
    private final ServerCapabilities capabilities;
    // Retains original expressions that were pushed server-side
    private final Map<SchemaPath, String> pushedFilters = new HashMap<>();
    private final Map<SchemaPath, String> pushedSorts = new HashMap<>();

    public String rewrite(SchemaContext ctx, LayoutQueryState queryState) {
        // 1. For each path with filter/sort in queryState:
        //    Check if pushable, translate, add to AST
        //
        // 2. Null pushed expressions on queryState inside suppressNotifications:
        queryState.suppressNotifications(() -> {
            for (var path : newlyPushedFilters) {
                pushedFilters.put(path, queryState.getFieldState(path).filterExpression());
                queryState.setFilterExpression(path, null);
            }
            for (var path : newlyPushedSorts) {
                pushedSorts.put(path, queryState.getFieldState(path).sortFields());
                queryState.setSortFields(path, null);
            }
        });
        //
        // 3. On next cycle, check if user cleared a pushed expression:
        //    If pushedFilters.containsKey(path) && queryState filter is null
        //    && original was not null -> user cleared it, remove from pushed map
        //
        // 4. Delegate to QueryBuilder.reconstruct()
    }
}
```

The suppressNotifications batch fires exactly one change notification after all nulls are applied. The fetchInProgress guard in AutoLayoutController prevents this notification from triggering a second fetch.

### Re-entrancy sequence:

```
1. User sets filter on path P
2. Change listener fires (fetchInProgress=false)
3. fetchInProgress = true
4. queryRewriter.rewrite():
   a. Reads filter from queryState
   b. Pushes to server AST
   c. suppressNotifications: nulls filter on queryState
   d. suppressNotifications exits -> fires change listener
   e. Change listener: fetchInProgress==true, returns immediately
5. rewrite() returns rewritten query
6. fetchAsync(rewritten) -> background thread
7. Result -> Platform.runLater -> setData() -> fetchInProgress=false
```

### Test Strategy

- **QueryRewriterDoubleEvalTest**:
  - Rewrite with pushable filter, verify queryState.getFieldState(path).filterExpression() is null after rewrite
  - Verify pushedExpressions retains original value
  - Call rewrite again with same state, verify re-push uses retained expression
  - Simulate user clearing filter (set null), verify pushed map is cleaned up
  - Verify non-pushable expressions are NOT nulled
- **EndToEndRewriteTest**:
  - Stub HTTP server on port 0 returning mock GraphQL data
  - Full lifecycle: construct controller -> set endpoint -> set query -> apply filter interaction -> verify rewritten query sent to stub -> verify response rendered
  - Verify no double evaluation by checking that client-side expression pipeline sees null for pushed expressions

### Execution Instructions

1. Write `QueryRewriterDoubleEvalTest.java` first (TDD)
2. Add `pushedExpressions` state to `QueryRewriter`
3. Implement null-expression push-down in `rewrite()`
4. Write `EndToEndRewriteTest.java`
5. Verify: `mvn -pl kramer-ql test` and `mvn -pl explorer test`
6. Run full build: `mvn clean install`

## Risk Factors

| Risk | Severity | Mitigation |
|------|----------|------------|
| Argument naming varies across GraphQL implementations | High | Fixed recognized names in v1; configurable in follow-on RDR |
| Introspection disabled on some endpoints | Medium | Graceful fallback to `ServerCapabilities.NONE` (all client-side) |
| Push-down produces different results than client-side | Medium | Test equivalence for simple filter/sort cases |
| Re-entrancy from suppressNotifications change notification | High | fetchInProgress guard tested explicitly in Phase 3b |
| LayoutQueryState.overriddenPaths() cross-module addition | Low | Single-line accessor, no behavior change |
| JavaFX testing complexity for Phases 3a/3b/3c | Medium | Follow GestureIntegrationTest pattern with Platform.startup() |
