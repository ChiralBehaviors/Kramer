# RDR-014 Implementation Plan: Conditional Display & Data Filtering

**Epic**: Kramer-7up
**RDR**: [RDR-014](../rdr/RDR-014-conditional-display-data-filtering.md) (accepted 2026-03-15)
**Date**: 2026-03-15

## Executive Summary

Two features from SIEUFERD (Bakke et al., SIGMOD 2016):

1. **Stable Tuple-Identifying Sort** -- deterministic row ordering by sortFields with type-aware comparison
2. **HIDE PARENT IF EMPTY** -- filter parents whose child Relations all produce empty arrays

Delivered as two sequential PRs. PR1 (sort) has no blockers. PR2 (hide-if-empty) depends on PR1 being merged. Both features modify `RelationLayout.measure()` and `extractFrom()` with pipeline order: sort -> filter -> measure children.

### Task 2: Kramer-090 -- Implement stable sort in RelationLayout

**Files**:
- `kramer/src/main/java/com/chiralbehaviors/layout/RelationLayout.java`

**Test** (TDD first):
- `kramer/src/test/java/com/chiralbehaviors/layout/RelationLayoutSortTest.java`
- `mvn -pl kramer test -Dtest=RelationLayoutSortTest`

**Changes in `measure()`** (after line 398, before child iteration):
1. Resolve sort fields: explicit `sortFields` > autoSort heuristic > empty (no-op)
2. Auto-detect heuristic: fields named `id`, `key`, `name` in priority order; fallback to first Primitive child
3. Sort `datum` ArrayNode using `buildComparator(resolvedSortFields)`
4. Store resolved sort fields on `RelationLayout` instance (new field alongside `extractor`)
5. `maxCardinality = sortedItems.size()`

**Changes in `extractFrom()`** (line 274-276):
1. After `extractor.apply(datum)`, apply same sort using stored resolved fields
2. Return sorted result to `node.extractFrom()`

**Type-aware comparator**:
- Numeric fields: `Double.compare(a.asDouble(), b.asDouble())`
- Text fields: `a.asText().compareTo(b.asText())`
- Null/missing: sort last (return +1 vs non-null)

**Acceptance criteria**:
- [x] Numeric IDs sort numerically (2 < 9 < 10, not 10 < 2 < 9)
- [x] autoSort=false: no sort applied
- [x] autoSort=true with no explicit sortFields: picks `id` field
- [x] Null fields sort last without NPE
- [x] Sort in both measure() and extractFrom()
- [x] All existing `RelationComputeTest` tests pass

### Task 5: Kramer-xa0 -- Implement hide-if-empty filter in RelationLayout

**Files**:
- `kramer/src/main/java/com/chiralbehaviors/layout/RelationLayout.java`

**Test** (TDD first):
- `kramer/src/test/java/com/chiralbehaviors/layout/RelationLayoutHideIfEmptyTest.java`
- `mvn -pl kramer test -Dtest=RelationLayoutHideIfEmptyTest`

**Combined pipeline in `measure()`**: sort -> filter -> measure children

**Changes in `measure()`** (after sort step, before child iteration):
```java
boolean resolvedHideIfEmpty = resolveHideIfEmpty(/*stylesheet*/, /*path*/);
boolean shouldFilter = resolvedHideIfEmpty && getNode().getAutoFoldable() == null;
if (shouldFilter) {
    List<JsonNode> filtered = StreamSupport.stream(items.spliterator(), false)
        .filter(item -> hasNonEmptyChildren(item, getNode()))
        .toList();
    items = JsonNodeFactory.instance.arrayNode().addAll(filtered);
}
maxCardinality = items.size(); // AFTER filtering
```

**Changes in `extractFrom()`**: compose filter with extractor, same as sort composition.

**`hasNonEmptyChildren` semantics** (shallow):
- Returns true if ANY child Relation has non-empty array data
- Parents with only Primitive children: never filtered (return true)
- All child Relations empty: return false

**autoFold guard**: `shouldFilter` is false when `getNode().getAutoFoldable() != null` because `flatten()` destroys parent-child structure.

**Acceptance criteria**:
- [x] hideIfEmpty=false: all parents rendered, maxCardinality = JSON array size
- [x] hideIfEmpty=true: empty-children parents excluded
- [x] autoFoldable guard prevents filtering
- [x] Primitive-only parents never filtered
- [x] Both measure() and extractFrom() consistent
- [x] maxCardinality=0 when all filtered (phantom row deferred to Phase 2)
- [x] All existing tests pass: `mvn -pl kramer test`

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Build-path bypass (extractFrom sees different data than measure) | Medium | Each task explicitly requires both paths; integration tests verify consistency |
| autoFold interaction | Medium | Explicit guard; test case 9 verifies |
| RDR-013 cache invalidation | Low | Cross-reference: stylesheet version counter (Kramer-63n) must be in place before hide-if-empty can be toggled at runtime via stylesheet override |
| Phantom row when all items filtered | Low | Deferred to Phase 2; documented in RDR risk table |

## Cross-RDR Dependencies

- **RDR-013 (Kramer-63n)**: Stylesheet version counter must increment on `setOverride()` to invalidate cached `MeasureResult` when `hide-if-empty` or `sort-fields` are toggled. Not a blocker for PR1/PR2 implementation but required before runtime toggling via stylesheet is safe.
- **RDR-009**: `LayoutStylesheet.getBoolean()` already implemented. No additional RDR-009 work needed.
- **RDR-015/016**: Both sort and filter feed the combined pipeline; later RDRs extend rather than replace.
