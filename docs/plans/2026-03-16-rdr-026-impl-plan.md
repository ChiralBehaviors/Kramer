# RDR-026 Implementation Plan: QueryState — Layer 2 Query Model

- **RDR**: 026
- **Date**: 2026-03-16
- **Status**: planned

---

## Executive Summary

Implement `QueryState`, a typed `LayoutStylesheet` implementation that represents Layer 2 (Query Model) of Kramer's 7-layer architecture. `QueryState` wraps `DefaultLayoutStylesheet`, adds typed setters for 9 field properties, change listener support, serialization, and `reset()`. It becomes the single source of truth for user-intent overrides, eliminating raw string-key manipulation.

New Java package: `com.chiralbehaviors.layout.query` (in `kramer` module).

Zero new external dependencies. Jackson (already present) used for serialization.

### FieldState Record

```java
public record FieldState(
    Boolean visible,            // default: true  (LayoutPropertyKeys.VISIBLE)
    String renderMode,          // default: null   (LayoutPropertyKeys.RENDER_MODE)
    Boolean hideIfEmpty,        // default: false  (LayoutPropertyKeys.HIDE_IF_EMPTY)
    String sortFields,          // default: null   (LayoutPropertyKeys.SORT_FIELDS)
    String filterExpression,    // default: null   (LayoutPropertyKeys.FILTER_EXPRESSION)
    String formulaExpression,   // default: null   (LayoutPropertyKeys.FORMULA_EXPRESSION)
    String aggregateExpression, // default: null   (LayoutPropertyKeys.AGGREGATE_EXPRESSION)
    String sortExpression,      // default: null   (LayoutPropertyKeys.SORT_EXPRESSION)
    String pivotField           // default: null   ("pivot-field")
) {}
```

9 properties. All nullable — `null` means "use default / not set."

### QueryState Design

```java
public class QueryState implements LayoutStylesheet {
    private final DefaultLayoutStylesheet inner;
    private final Map<SchemaPath, FieldState> fieldStates = new HashMap<>();
    private final List<Runnable> changeListeners = new ArrayList<>();
    private boolean suppressNotifications = false;

    public QueryState(Style style) {
        this.inner = new DefaultLayoutStylesheet(style);
    }

    // --- LayoutStylesheet delegation (7 methods) ---
    // getVersion, getDouble, getInt, getString, getBoolean,
    // primitiveStyle, relationStyle → all delegate to inner

    // --- Typed setters (9 methods) ---
    // setVisible(SchemaPath, Boolean)
    // setRenderMode(SchemaPath, String)
    // setHideIfEmpty(SchemaPath, Boolean)
    // setSortFields(SchemaPath, String)
    // setFilter(SchemaPath, String)
    // setFormula(SchemaPath, String)
    // setAggregate(SchemaPath, String)
    // setSortExpression(SchemaPath, String)
    // setPivotField(SchemaPath, String)
    //
    // Each: updates fieldStates map, calls inner.setOverride(), fires listeners

    // --- Query methods ---
    // getFieldState(SchemaPath) → FieldState (null fields = not set)
    // getVisibleOrDefault(SchemaPath) → boolean (never null, defaults to true)

    // --- Lifecycle ---
    // reset() → clears fieldStates, calls inner.clearOverrides(), fires listeners
    // addChangeListener(Runnable)
    // removeChangeListener(Runnable)
}
```

**Key design decision**: QueryState maintains its own `Map<SchemaPath, FieldState>` for tracking explicit user settings. This is NOT dual representation — `DefaultLayoutStylesheet` does not expose which properties have been set vs defaulted (getBoolean returns a default, indistinguishable from an explicit set to the same value). The fieldStates map tracks **intent metadata**; the inner stylesheet handles all layout reads.

### Setter Pattern

Each typed setter follows this pattern:

```java
public void setFilter(SchemaPath path, String expression) {
    var current = fieldStates.getOrDefault(path, EMPTY);
    fieldStates.put(path, new FieldState(
        current.visible(), current.renderMode(), current.hideIfEmpty(),
        current.sortFields(), expression, current.formulaExpression(),
        current.aggregateExpression(), current.sortExpression(), current.pivotField()));
    if (expression != null) {
        inner.setOverride(path, LayoutPropertyKeys.FILTER_EXPRESSION, expression);
    }
    // Note: setting to null should remove the override (needs clearOverride or re-apply)
    fireListeners();
}
```

For null-setting (removing an override): since `DefaultLayoutStylesheet` lacks `removeOverride()`, the simplest approach is to call `inner.clearOverrides()` and re-apply all non-null field states. An alternative is adding `removeOverride(SchemaPath, String)` to `DefaultLayoutStylesheet`. The implementer should evaluate both — the re-apply approach is correct but O(n); the removeOverride approach is O(1) but requires modifying `DefaultLayoutStylesheet`.

## Phase 2: Serialization (Kramer-430w)

**Blocked by**: Kramer-fmfy (Phase 1)

**Goal**: Add `toJson()` / `fromJson()` to `QueryState` for persisting user configurations.

### Files to Modify/Create (TDD Order)

1. `QueryStateSerializationTest.java` — new test file (or extend `QueryStateTest.java`)
2. `QueryState.java` — add `toJson()` and `fromJson()` methods

### Serialization Format

```json
{
  "orders": {
    "visible": false,
    "hideIfEmpty": true
  },
  "orders/amount": {
    "filterExpression": "$value > 100",
    "sortExpression": "$value"
  }
}
```

- Top-level keys: `SchemaPath.toString()` (slash-delimited segments)
- Values: JSON object with non-null `FieldState` fields only
- Field names match `FieldState` record component names (camelCase)

### Key Behaviors

- `toJson()`: Returns Jackson `ObjectNode`. Iterates `fieldStates` map, emits only non-null fields per path.
- `fromJson(ObjectNode, Style)`: Static factory. Creates new `QueryState`, parses JSON, populates via setters with `suppressNotifications = true`. Fires exactly one change notification on completion.
- Unknown JSON fields are silently ignored (forward compatibility).
- Empty JSON (`{}`) produces an empty QueryState with no overrides.

### SchemaPath Reconstruction

`SchemaPath.toString()` produces `"segment1/segment2"`. Reconstruction requires `new SchemaPath("segment1", "segment2")` (splitting on `/`). Verify `SchemaPath` has a varargs or List constructor that accepts segments.

From the symbol analysis: `SchemaPath` has constructors `SchemaPath[0]`, `SchemaPath[1]`, `SchemaPath[2]`. The record has `segments` as its component (a `List<String>`). The canonical constructor takes `List<String>` and there are convenience constructors.
