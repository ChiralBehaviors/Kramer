## Phase 1: Scroll Preservation in Outline (LOW effort)

**Bead**: Kramer-mdl
**Priority**: P2
**Blockers**: none

**Problem**: `Outline.updateItem()` unconditionally calls `showAsFirst(0)` on every data update (line 77-86), resetting scroll to top. `NestedTable.updateItem()` does not have this bug.

**Files**:
- `kramer/src/main/java/com/chiralbehaviors/layout/outline/Outline.java` (MODIFY lines 77-86)
- `kramer/src/test/java/com/chiralbehaviors/layout/outline/OutlineScrollPreservationTest.java` (NEW — TDD first)

**TDD**:
1. Test: populate Outline, scroll to row N via `showAsFirst(N)`, call `updateItem()` with same-size data, assert `getFirstVisibleIndex()` returns value >= N (clamped to new size).
2. Test: first population (no prior visible cells, `getFirstVisibleIndex()` returns empty), assert `showAsFirst(0)` behavior preserved.
3. Test: size-decreasing update — assert index clamped to `items.size() - 1`.

**Implementation**:
```java
// Before:
if (!items.isEmpty()) {
    showAsFirst(0);
}

// After:
OptionalInt savedIndex = getFirstVisibleIndex();
items.setAll(list);
if (!items.isEmpty()) {
    savedIndex.ifPresentOrElse(
        idx -> showAsFirst(Math.min(idx, items.size() - 1)),
        () -> showAsFirst(0)
    );
}
```

**Test command**: `mvn -pl kramer test -Dtest=OutlineScrollPreservationTest`

## Phase 2: List Diffing for Data Updates (MEDIUM effort)

**Bead**: Kramer-h7q
**Priority**: P2
**Blockers**: none (independent of Phase 1)

**Problem**: `items.setAll()` fires a bulk `LIST_UPDATED` event causing all visible cells to re-invoke cellFactory, even for unchanged rows. VirtualFlow recycles cells but cannot skip unchanged rows without per-element equality.

**Files**:
- `kramer/src/main/java/com/chiralbehaviors/layout/outline/Outline.java` (MODIFY updateItem)
- `kramer/src/main/java/com/chiralbehaviors/layout/table/NestedTable.java` (MODIFY updateItem)
- `kramer/src/test/java/com/chiralbehaviors/layout/ListDiffUpdateTest.java` (NEW — TDD first)

**TDD**:
1. Test: same-size update with 1 changed row — cell factory invoked exactly once (instrument with AtomicInteger counter wrapping cellFactory).
2. Test: size-mismatch update — falls back to `setAll()`, all cells updated.
3. Test: all-same update — zero cell factory invocations.
4. Benchmark test: 100-row update with 5 changed rows completes in < 2ms (use `System.nanoTime()`).

**Implementation** (identical change in both `Outline.updateItem()` and `NestedTable.updateItem()`):
```java
List<JsonNode> incoming = SchemaNode.asList(item);
if (incoming.size() == items.size()) {
    for (int i = 0; i < incoming.size(); i++) {
        if (!incoming.get(i).equals(items.get(i))) {
            items.set(i, incoming.get(i));
        }
    }
} else {
    items.setAll(incoming);
}
```

**Test command**: `mvn -pl kramer test -Dtest=ListDiffUpdateTest`

## Phase 3 Prerequisite: SchemaPath Derivation at Construction (MEDIUM effort)

**Bead**: Kramer-e1l
**Priority**: P2
**Blockers**: none (standalone deliverable)

**Problem**: `SchemaPath` is currently set as a side-effect of `measure()` (see `RelationLayout.measure()` ~line 400-407 and `AutoLayout.measure()` ~line 135). When Phase 3 cache bypasses measure on a hit, `SchemaPath` will be null, breaking all cache key lookups. This is also a prerequisite for RDR-013, 014, 015, 017, 018, 019 (all use SchemaPath-addressed LayoutStylesheet lookups).

- `kramer/src/main/java/com/chiralbehaviors/layout/AutoLayout.java` (MODIFY ~line 135)
- `kramer/src/main/java/com/chiralbehaviors/layout/schema/SchemaNode.java` (possibly MODIFY)
- `kramer/src/test/java/com/chiralbehaviors/layout/SchemaPathDerivationTest.java` (NEW — TDD first)

**TDD**:
1. Test: construct `RelationLayout` from a `Relation`; assert `schemaPath` is non-null before any `measure()` call.
2. Test: root `Relation` has path equal to its `SchemaNode.name`.
3. Test: nested `Relation` (child of root) has path `"root.child"`.
4. Test: after derivation at construction, `measure()` does NOT overwrite SchemaPath with a different value.

**Implementation**: Derive `SchemaPath` in `SchemaNodeLayout` constructor (or `SchemaNode.buildLayout()` factory) by walking the `SchemaNode` tree, not as a measure side-effect.

## Phase 3: Layout Decision Caching for Resize (MEDIUM effort)

**Bead**: Kramer-6g1
**Priority**: P2
**Blockers**: Kramer-e1l (SchemaPath at construction)

**Problem**: `AutoLayout.resize()` always runs `layout()` + `compress()` + `buildControl()` even when the layout decision is unchanged. Only `buildControl()` must run (output is width-dependent); the decision phase can be cached.

**Files**:
- `kramer/src/main/java/com/chiralbehaviors/layout/AutoLayout.java` (MODIFY resize path + add root property listener)
- `kramer/src/main/java/com/chiralbehaviors/layout/LayoutDecisionKey.java` (NEW record)
- `kramer/src/test/java/com/chiralbehaviors/layout/LayoutDecisionCacheTest.java` (NEW — TDD first)

**LayoutDecisionKey record**:
```java
record LayoutDecisionKey(SchemaPath path, int widthBucket, int dataCardinality) {
    static LayoutDecisionKey of(SchemaPath path, double width, int itemCount) {
        return new LayoutDecisionKey(path, (int)(width / 10), itemCount);
    }
}
```

**Cache in AutoLayout**:
```java
Map<LayoutDecisionKey, LayoutResult> decisionCache = new HashMap<>();
```

**Invalidation rules** (all four must be implemented and tested):
1. Stylesheet listener fires → `decisionCache.clear()` (already nulls `layout`)
2. `autoLayout()` public method called → `decisionCache.clear()` (resets `layoutWidth=0`)
3. Data cardinality changes → key miss, natural recompute (no explicit invalidation needed)
4. Root (`SchemaNode`) property changes → new listener mirrors stylesheet listener: null `layout`, null `measureResult`, clear `decisionCache`

**TDD**:
1. Test: resize to same width bucket, same cardinality → `layout()` + `compress()` NOT called (instrument with AtomicInteger spy).
2. Test: resize to different width bucket → cache miss, full pipeline runs.
3. Test: cardinality change (add a row) → cache miss, full pipeline runs.
4. Test: `autoLayout()` call clears cache (next resize runs full pipeline).
5. Test: stylesheet change clears cache.
6. Test: `setRoot()` call nulls layout and clears cache.

**Test command**: `mvn -pl kramer test -Dtest=LayoutDecisionCacheTest`

## Phase 4: Convergent Width Integration with RDR-013 (LOW effort)

**Bead**: Kramer-n9w
**Priority**: P2
**Blockers**: Kramer-16k (RDR-013 P2-T2: Convergence detection in PrimitiveLayout)

**Problem**: After Phase 3, resize still runs `buildControl()` on every width change. Once all `ContentWidthStats` are converged (from RDR-013), column widths cannot change — only cell content updates. Settled layouts can skip `layout()` + `compress()` unconditionally on resize.

**Files**:
- `kramer/src/main/java/com/chiralbehaviors/layout/AutoLayout.java` (MODIFY resize path)
- `kramer/src/test/java/com/chiralbehaviors/layout/ConvergentWidthStabilityTest.java` (NEW — TDD first)

**Design**:
```java
boolean isSettled = allColumns.stream().allMatch(col -> col.widthStats().converged());
if (isSettled && sameWidthBucket && sameCardinality) {
    buildControl(layout, data);   // skip layout()+compress()
    return;
}
// else full pipeline
```

**Stability guarantee**: once converged, new data cannot change column widths. First N data updates may adjust widths (statistical convergence); after convergence, widths frozen, only cell content updates fire.

**TDD** (after RDR-013 merged):
1. Test: all `ContentWidthStats.converged==true` → resize skips `layout()` + `compress()`.
2. Test: any column unconverged → full pipeline still runs.
3. Test: data update after convergence → only `buildControl()` fires.

**Test command**: `mvn -pl kramer test -Dtest=ConvergentWidthStabilityTest`

3. Resize with same width bucket skips layout decision phase (Phase 3).
4. Column widths stable after convergence (Phase 4 + RDR-013).
5. No visual artifacts from stale layout decisions.
6. List-diff update < 2ms for 100-row change (Phase 2).
7. All existing tests pass.
